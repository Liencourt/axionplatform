Axiom Platform — API Reference

Versão v1.0  ·  Base URL: /api/v1/  ·  Protocolo: HTTPS

OpenAPI Schema
Autenticação
Header obrigatório em todas as requisições
Segurança: Nunca exponha sua API Key em código front-end, repositórios públicos ou logs. Em caso de comprometimento, contate o suporte para rotação imediata.

Header obrigatório

X-Axiom-API-Key: xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx

Obtenha sua chave em Minha Conta → Chave de API (B2B). A chave é UUID v4, única por empresa e rotacionável mediante solicitação.

Formato de Resposta Padrão
{ "status": "sucesso" | "erro" | "aviso", "mensagem": "Descrição legível", "kpis": { /* métricas calculadas */ }, "data": [ /* registros detalhados */ ] }
200 Sucesso 201 Criado 207 Parcial 400 Dado inválido 401 API Key inválida 404 Não encontrado

207 Parcial — ocorre quando parte do batch foi rejeitada. O campo data[] lista os erros linha a linha (até 50 detalhes).

Modelo Multi-Loja (Filiais)
Conceito: Um projeto pertence à empresa, mas dados são segregados por filial (loja). Elasticidade e forecast são calculados de forma independente por filial quando especificada.

Fluxo recomendado

  1. Registre as filiais: POST /api/v1/lojas/
  2. Crie projeto por escopo: POST /api/v1/projetos/
  3. Envie vendas com loja_id: POST /api/v1/ingestao/vendas/
  4. Rode a IA no painel (Construtor de Hipóteses)
  5. Consulte elasticidade por filial: GET /api/v1/elasticidade/{id}/?loja_id=X

Garantias de segurança (anti-IDOR)

  • Loja validada contra a empresa da API Key
  • Projetos de outra empresa retornam 404
  • Todos os queries filtram por empresa=request.empresa
  • IDs cross-tenant são impossíveis de adivinhar (UUID)
Endpoints de Ingestão (ERP → Axiom)
G/P
/api/v1/lojas/
Gerencia filiais da empresa autenticada

GET — Listar filiais

# Resposta 200 { "status": "sucesso", "data": [ { "id": 1, "nome": "Filial RJ-Centro", "ativo": true }, { "id": 2, "nome": "Filial SP-Paulista", "ativo": true } ] }

POST — Criar filial

# Body { "nome": "Filial SP-Paulista" } # Resposta 201 (ou 200 se já existe) { "data": [{ "id": 3, "nome": "Filial SP-Paulista" }] }

Parâmetros POST

CampoTipoDescrição
nomestringObrig.Nome da filial. Idempotente — retorna existente se já cadastrada.
Guarde o id retornado. Ele é o loja_id usado em todos os outros endpoints.
G/P
/api/v1/projetos/
Cria e lista projetos de precificação

POST — Criar projeto

# Body { "nome": "Pricing Filial RJ — Q1 2026", "loja_id": 1 } # Resposta 201 { "data": [{ "projeto_id": 12, "nome": "Pricing Filial RJ — Q1 2026", "loja": "Filial RJ-Centro" }] }

GET — Query string

ParâmetroDescrição
?loja_id=3Filtra projetos de uma filial.

Parâmetros POST

CampoTipoDescrição
nomestringObrig.Nome do projeto.
loja_idintOpcionalID da filial. Omita para projeto global da empresa.
loja_nomestringOpcionalAlternativa ao loja_id — busca por nome.
POST
/api/v1/ingestao/vendas/
Batch de vendas por SKU → alimenta elasticidade e Axiom Forecast Micro
Limite de volume: Máx. 5.000 registros por requisição. Para volumes maiores, fragmente em múltiplos POSTs. Registros duplicados (empresa + loja + SKU + data) são atualizados automaticamente (upsert).

Body (JSON)

{ "projeto_id": 12, "loja_id": 1, "registros": [ { "codigo_produto": "SKU001", "nome_produto": "Arroz Tipo 1 5kg", "data_venda": "2025-12-31", "quantidade": 120, "preco_praticado": 12.90, "custo_unitario": 8.20, "variaveis_extras": { "feriado": true, "temperatura": 28.5 } } ] }

Campos de cada registro

CampoTipoDescrição
codigo_produtostringObrig.Código SKU.
data_vendastringObrig.Formato YYYY-MM-DD.
quantidadenumberObrig.Unidades vendidas. Deve ser > 0.
preco_praticadonumberObrig.Preço de venda em R$. Deve ser > 0.
nome_produtostringOpcionalNome legível do produto.
custo_unitarionumberOpcionalCusto unitário em R$. Necessário para margem.
variaveis_extrasobjectOpcionalCovariáveis livres: feriado, clima, estoque, etc.

Resposta

{ "kpis": { "recebidos": 150, "inseridos": 140, "atualizados": 10, "rejeitados": 0 }, "data": [] // erros linha a linha se rejeitados > 0 }
POST
/api/v1/ingestao/faturamento/
Faturamento diário consolidado → Axiom Macro Forecast (Prophet)
Diferença de /vendas/: Recebe apenas a receita total do dia sem detalhamento por SKU. Use para o motor Prophet (previsão macro de faturamento). Para previsão por produto, use /ingestao/vendas/.

Body (JSON)

{ "loja_id": 1, // null = empresa consolidada "registros": [ { "data": "2025-12-29", "faturamento": 58420.50 }, { "data": "2025-12-30", "faturamento": 62100.00 }, { "data": "2025-12-31", "faturamento": 71380.75 } ] }

Campos de cada registro

CampoTipoDescrição
datastringObrig.Formato YYYY-MM-DD.
faturamentonumberObrig.Receita total do dia em R$. Deve ser ≥ 0.
Upsert: Enviar o mesmo dia novamente atualiza o valor. Ideal para corrigir fechamentos sem precisar de DELETE.

Exemplos de Integração
Python — Fechamento Diário Automatizado
import requests from datetime import date AXIOM_KEY = "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx" BASE = "https://seudominio.com/api/v1" HDR = {"X-Axiom-API-Key": AXIOM_KEY, "Content-Type": "application/json"} # 1. Garantir que as filiais existam r = requests.post(f"{BASE}/lojas/", json={"nome": "Filial RJ-Centro"}, headers=HDR) loja_id = r.json()["data"][0]["id"] # 2. Criar (ou reusar) projeto r = requests.get(f"{BASE}/projetos/?loja_id={loja_id}", headers=HDR) projetos = r.json()["data"] if projetos: projeto_id = projetos[0]["projeto_id"] else: r = requests.post(f"{BASE}/projetos/", json={"nome": "Pricing 2026", "loja_id": loja_id}, headers=HDR) projeto_id = r.json()["data"][0]["projeto_id"] # 3. Enviar batch de vendas (fragmentado se > 5000) vendas_do_dia = [ # ← sua query no ERP {"codigo_produto": "SKU001", "data_venda": str(date.today()), "quantidade": 120, "preco_praticado": 12.90, "custo_unitario": 8.20}, ] LOTE = 5000 for i in range(0, len(vendas_do_dia), LOTE): r = requests.post(f"{BASE}/ingestao/vendas/", headers=HDR, json={ "projeto_id": projeto_id, "loja_id": loja_id, "registros": vendas_do_dia[i:i+LOTE] }) k = r.json()["kpis"] print(f"Lote {i//LOTE+1}: +{k['inseridos']} / ~{k['atualizados']} / ✗{k['rejeitados']}") # 4. Enviar faturamento macro requests.post(f"{BASE}/ingestao/faturamento/", headers=HDR, json={ "loja_id": loja_id, "registros": [{"data": str(date.today()), "faturamento": 58420.50}] })
Node.js — Integração com paginação automática
const AXIOM_KEY = 'xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx'; const BASE = 'https://seudominio.com/api/v1'; const HDR = { 'X-Axiom-API-Key': AXIOM_KEY, 'Content-Type': 'application/json' }; async function enviarVendasPaginado(projetoId, lojaId, todosRegistros) { const LOTE = 5000; let totalInseridos = 0, totalRejeitados = 0; for (let i = 0; i < todosRegistros.length; i += LOTE) { const lote = todosRegistros.slice(i, i + LOTE); const res = await fetch(`${BASE}/ingestao/vendas/`, { method: 'POST', headers: HDR, body: JSON.stringify({ projeto_id: projetoId, loja_id: lojaId, registros: lote }) }); const { kpis, data: erros } = await res.json(); totalInseridos += kpis.inseridos; totalRejeitados += kpis.rejeitados; if (erros.length) console.warn('Erros:', erros); } console.log(`✓ Total: +${totalInseridos} inseridos, ✗${totalRejeitados} rejeitados`); }
